## 🎯 خطة التطوير المقترحة (3 مراحل)

### المرحلة 1: تحسين UI + شروحات (Frontend فقط) 🟢

> **لا يحتاج تغييرات في الـ Database**

1. **إضافة شروحات واضحة** في كل تبويب ليفهم المستخدم:
   - تبويب المراحل: شرح ما هي المرحلة الابتدائية/النهائية + أمثلة
   - تبويب التحويلات: شرح بصري "من أين إلى أين" + ما معنى الأدوار
   - تبويب الإشعارات: شرح كل قناة + كيف يتم الربط
   - وبانر متكامل في أعلى كل تبويب

2. **تطوير ديالوج التحويلات** ليشمل:
   - اختيار الأدوار المسموحة بـ multi-select chips (بدل hardcoded)
   - تفعيل/تعطيل `requires_comment` مع توضيح
   - تفعيل/تعطيل `requires_approval` مع اختيار `approval_roles`

3. **تحسين عرض المسار (Flow View)**:
   - إضافة أيقونات الموافقة/الرفض على الأسهم
   - عرض الأدوار المسموحة على كل مسار
   - تلوين مسارات الموافقة بلون مختلف عن مسارات الرفض

4. **لوحة معلومات سريعة** في أعلى الصفحة:
   - عدد المراحل / الانتقالات / الإشعارات المفعّلة
   - آخر تعديل

### المرحلة 2: نظام الموافقات + تحديد المستلمين 🟡

> **يحتاج تغييرات بسيطة في الـ Database**

1. **تحديد مستلمي الإشعار**:
   - إضافة UI لاختيار المستلمين (أدوار / مستخدمين محددين)
   - ربطه بجدول `workflow_notification_settings` (حقل `recipients` JSONB موجود)

2. **نظام الموافقات الفعلي**:
   - عند `requires_approval = true`:
     - يتوقف المستند عند حالة "في انتظار الموافقة"
     - ُرسل إشعار للمسؤولين المحددين
     - يظهر زر قبول/رفض مع حقل تعليق
   - جدول `workflow_approvals` جديد لتسجيل الموافقات/الرفض

3. **Approval Log** في كل مستند:
   - سجل من وافق/رفض + متى + التعليق

### المرحلة 3: الشروط الديناميكية + الإجراءات التلقائية 🔴

> **تطوير متقدم**

1. **شروط مبنية على المستند**:
   - مثل: "إذا كان المبلغ > 10,000 → يحتاج موافقة مدير"
   - Condition Builder بسيط في UI

2. **إجراءات تلقائية** (auto_actions):
   - إنشاء مستند تالٍ تلقائياً
   - إرسال بريد/تلغرام
   - تحديث حقول المستند
