# 🏗️ نظام الحسابات الفرعية للأطراف (Sub-Ledger)
## Party Sub-Account System — Implementation Document

**تاريخ التنفيذ:** 2026-02-23  
**الحالة:** ✅ مكتمل — يعمل لجميع المشتركين

---

## المشكلة السابقة

كل الموردين والعملاء كانوا مربوطين **بنفس الحساب** في الشجرة المحاسبية:
- كل الموردين → `2111 دين الموردين - الأقمشة` (حساب واحد!)
- كل العملاء → `1131 ذمم الجملة` (حساب واحد!)

**النتيجة:** الرصيد يظهر خاطئ لكل مورد (رصيد واحد مشترك) والتقارير غير دقيقة.

---

## الحل المُنفذ

### الهيكل الجديد:
```
2111 — دين الموردين (مجموعة GROUP — لا يُقيَّد عليه مباشرة)
├── 2111-001 — نانسي طقم (is_party_account = true)
├── 2111-002 — مصانع بورصة (is_party_account = true)
├── 2111-003 — مصانع قوانغدونغ (is_party_account = true)
└── ...

1131 — ذمم الجملة (مجموعة GROUP)
├── 1131-001 — ساشا ملدفان (is_party_account = true)
├── 1131-002 — مصنع الزهراء (is_party_account = true)
└── ...
```

### ما تم تنفيذه:

#### 1. قاعدة البيانات
| التغيير | التفاصيل |
|---------|---------|
| أعمدة جديدة | `chart_of_accounts.is_party_account`, `party_id`, `party_type` |
| فهرس | `idx_coa_party` على (`party_id`, `party_type`) |
| بيانات | 6 حسابات موردين + 7 حسابات عملاء مُنشأة |
| حسابات الأب | `2111` و `1131` تحولت من `is_detail` إلى `is_group` |
| قيود قائمة | تم تحريك سطور القيد لـ "نانسي طقم" للحساب الفرعي `2111-001` |

#### 2. Triggers تلقائية
| Trigger | الجدول | الشرط | الوظيفة |
|---------|--------|-------|---------|
| `trg_supplier_sub_account` | `suppliers` | `BEFORE INSERT` عندما `payable_account_id IS NULL` | يُنشئ حساب `2111-XXX` فرعي تلقائياً |
| `trg_customer_sub_account` | `customers` | `BEFORE INSERT` عندما `receivable_account_id IS NULL` | يُنشئ حساب `1131-XXX` فرعي تلقائياً |

**ما يحدث عند إضافة مورد جديد:**
1. المستخدم يُنشئ المورد (بدون تحديد حساب)
2. الـ Trigger يبحث عن حساب الأب (2111) في نفس الشركة
3. يُنشئ حساب فرعي بكود تسلسلي (2111-007 مثلاً)
4. يربط المورد بالحساب الجديد تلقائياً

#### 3. الفرونت إند
| الملف | التغيير |
|-------|---------|
| `accountsService.ts` | فلتر `.or('is_party_account.is.null,is_party_account.eq.false')` — يخفي الحسابات الفرعية من الشجرة |
| `SuppliersList.tsx` | الرصيد يُقرأ من `chart_of_accounts.current_balance` عبر `payable_account_id` |
| `SuppliersList.tsx` | النقر على المورد يفتح `UnifiedAccountingSheet` (docType='party') |

#### 4. نظام القيود (purchaseAccountingService.ts)
- **لم يتغير** — الكود يستخدم `supplier.payable_account_id` بالفعل
- بعد ربط كل مورد بحسابه الفرعي، القيود تُسجَّل تلقائياً في الحساب الصحيح

---

## كيف يعمل مع المشتركين الجدد

عند تسجيل مشترك جديد:
1. نظام الـ Provisioning يُنشئ الشجرة المحاسبية (بما فيها `2111` و `1131` كمجموعات)
2. عند إنشاء أول مورد → الـ Trigger يُنشئ `2111-001` تلقائياً
3. عند إنشاء أول عميل → الـ Trigger يُنشئ `1131-001` تلقائياً
4. الحسابات الفرعية **لا تظهر** في الشجرة المحاسبية
5. تظهر فقط في تبويبات الموردين/العملاء

## ⚠️ ملاحظات مهمة

1. **حسابات الأب (2111, 1131)** لا يجب القيد عليها مباشرة — فقط الحسابات الفرعية
2. **الترقيم تسلسلي** داخل كل شركة (2111-001, 2111-002, ...)
3. **رصيد حساب الأب** = مجموع أرصدة كل الحسابات الفرعية (يمكن حسابه بـ SQL)
4. **الـ is_party_account = true** يمنع ظهور الحساب في:
   - الشجرة المحاسبية الرئيسية
   - dropdown اختيار الحسابات
5. **Onboarding template** يجب أن يُنشئ `2111` و `1131` كمجموعات (`is_group = true`)
