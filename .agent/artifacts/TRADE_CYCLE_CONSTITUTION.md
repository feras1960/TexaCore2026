# 📜 دستور دورة التجارة الموحدة (Trade Cycle Constitution)

> **الوثيقة المرجعية الأساسية** لنظام المشتريات والمبيعات في TexaCore ERP
> آخر تحديث: 16 فبراير 2026

---

## 🏛️ المبدأ الأساسي: جدول واحد — مستند واحد — مراحل متعددة

### القاعدة الذهبية
```
❌ ممنوع: نسخ المستندات بين الجداول عند الانتقال بين المراحل
✅ واجب: تغيير عمود `stage` فقط — المستند يبقى في نفس الجدول
```

### البنية المعمارية

| الوحدة | الجدول الرئيسي | جدول الأصناف | ملاحظات |
|--------|----------------|-------------|---------|
| **المشتريات** | `purchase_transactions` | `purchase_transaction_items` | جميع مستندات الشراء |
| **المبيعات** | `sales_transactions` | `sales_transaction_items` | جميع مستندات البيع |
| **إذون التسليم** | `purchase_receipts` | `purchase_receipt_items` | تُنشأ من الفواتير المؤكدة |
| **المرتجعات** | `purchase_returns` | `purchase_return_items` | تُنشأ من الفواتير المستلمة |

### العمود المحوري: `stage`
كل مستند تجاري يحتوي على عمود `stage` (TEXT, NOT NULL, DEFAULT 'draft').
المستند يتحرك بين المراحل عبر تغيير هذا العمود فقط.
**عند الانتقال**: المستند يختفي من المرحلة السابقة ويظهر في المرحلة الجديدة.

## 📊 مراحل المشتريات (Purchase Stages)

### المراحل الأساسية (Core Stages)

```
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│  طلب     │──→│  عرض     │──→│  مسودة   │──→│  مؤكدة   │──→│  مستلمة  │──→│  مرحّلة  │
│ request  │   │quotation │   │  draft   │   │confirmed │   │ received │   │  posted  │
│          │   │          │   │          │   │          │   │          │   │          │
│ معلوماتي │   │ معلوماتي │   │ فاتورة  │   │ التزام   │   │  مخزون   │   │ محاسبة   │
│ DRAFT-XX │   │ DRAFT-XX │   │ DRAFT-XX │   │PI-000001 │   │PI-000001 │   │PI-000001 │
└──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
```

### تعريف كل مرحلة

| # | المرحلة | Stage Key | الوظيفة | الرقم | التأثير |
|---|---------|-----------|---------|-------|---------|
| 1 | **طلب شراء** | `request` | معلوماتي — "نحتاج هذه المواد" | `DRAFT-XXXXXX` | ❌ لا محاسبة ❌ لا مخزون |
| 2 | **عرض سعر** | `quotation` | معلوماتي — "المورد عرض هذا السعر" | `DRAFT-XXXXXX` | ❌ لا محاسبة ❌ لا مخزون |
| 3 | **مسودة فاتورة** | `draft` | تحضير الفاتورة | `DRAFT-XXXXXX` | ❌ لا محاسبة ❌ لا مخزون |
| 4 | **فاتورة مؤكدة** | `confirmed` | التزام رسمي تجاه المورد | ✅ `PI-2026-000001` | ✅ تظهر في استلامات معلقة |
| 5 | **مستلمة** | `received` | البضاعة وصلت المستودع | `PI-2026-000001` | ✅ حركة مخزون |
| 6 | **مرحّلة** | `posted` | قيد محاسبي مُسجّل | `PI-2026-000001` | ✅ قيد يومية |

### أعلام الحالة الإضافية (NOT separate stages — flags ON the invoice)

> **هذه ليست مراحل منفصلة بل أعلام تظهر على بطاقة الفاتورة**

| العَلَم | العمود في DB | القيم | يظهر كـ Badge |
|---------|-------------|-------|---------------|
| **الدفع** | `paid_amount` + `balance` + `total_amount` | حساب تلقائي | 🔴 غير مدفوعة / 🟡 مدفوعة جزئياً / 🟢 مدفوعة بالكامل |
| **التسليم** | `received_qty` vs `quantity` per item | حساب تلقائي | 📦 لم تُستلم / 📦 مستلمة جزئياً / ✅ مستلمة بالكامل |
| **الترحيل** | `is_posted` (boolean) | true/false | 📊 مرحّلة / — |

> **القاعدة**: الأعلام تظهر على البطاقة (في الجدول والكانبان) بجانب المرحلة الأساسية.
> لا نحتاج تبويبات منفصلة لحالة الدفع أو التسليم.

## 🔢 نظام الترقيم (Numbering System)

### القاعدة الأساسية
```
الرقم النهائي يُعطى فقط عند مرحلة CONFIRMED
قبلها = DRAFT-XXXXXX (مؤقت)
بعدها = الرقم لا يتغير أبداً
```

### بادئات الأرقام حسب نوع المستند

| نوع المستند | البادئة | مثال |
|-------------|---------|------|
| فاتورة شراء | `PI` | `PI-2026-000001` |
| فاتورة مبيعات | `SI` | `SI-2026-000001` |
| عرض سعر شراء | `PQ` | `PQ-2026-000001` |
| عرض سعر مبيعات | `SQ` | `SQ-2026-000001` |
| إذن تسليم | `GR` | `GR-2026-000001` |
| مرتجع شراء | `PR` | `PR-2026-000001` |
| مرتجع مبيعات | `SR` | `SR-2026-000001` |
| طلب شراء | `REQ` | `REQ-2026-000001` |

### تسلسل الأرقام
- **تسلسلي** (000001, 000002, 000003...) — ليس عشوائي
- **لكل شركة** (`company_id`) — مستقل
- **لكل سنة** — يبدأ من 000001 كل سنة جديدة
- **محمي من التكرار** — retry logic عند unique constraint violation

### التنفيذ الفني
- `TradeService.getNextReferenceNumber()` → يُرجع `DRAFT-XXXXXX` فقط
- `TradeService.assignPermanentNumber(docId, docType, companyId)` → يُعطي الرقم النهائي عند التأكيد
- `useSheetActions.ts` → `save_confirm` يستدعي `assignPermanentNumber`
- `useSheetActions.ts` → `unconfirm` يُرجع الرقم لـ `DRAFT-XXXXXX`

## 🔄 قواعد الانتقال بين المراحل (Transition Rules)

### المشتريات

| من | إلى | مسموح | الشروط | ما يحصل |
|----|-----|-------|--------|---------|
| `request` → `quotation` | ✅ | — | تغيير stage فقط |
| `request` → `draft` | ✅ | — | قفز مباشر للفاتورة |
| `quotation` → `draft` | ✅ | — | تحويل لفاتورة |
| `draft` → `confirmed` | ✅ | مورد + أصناف | **يُعطى رقم PI نهائي** |
| `confirmed` → `received` | ✅ | إذن تسليم مكتمل | حركة مخزون |
| `received` → `posted` | ✅ | — | قيد محاسبي |
| `confirmed` → `draft` | ✅ | لا يوجد استلام مربوط | يفقد رقم PI → DRAFT |
| `posted` → أي مرحلة | ❌ | — | **ممنوع — نهائي** |

### قواعد السحب والإفلات (Kanban Drag Rules)
- **يمكن السحب للأمام**: request → quotation → draft → confirmed
- **يمكن السحب للخلف**: quotation → request, draft → quotation/request
- **ممنوع السحب للخلف**: confirmed → draft (يحتاج زر "إلغاء تأكيد" صريح)
- **ممنوع نهائياً**: posted → أي مرحلة
- **عند السحب إلى confirmed**: يظهر dialog تأكيد + يُعطى رقم

## 🖥️ واجهة المستخدم (UI Layout)

### التبويبات في وضع الجدول (6 تبويبات فقط)

```
┌──────┬──────────────┬────────────┬────────────────┬──────────────┬───────────┐
│ الكل │ طلبات شراء   │ عروض سعر   │ فواتير مشتريات │ اذون تسليم  │ مرتجعات   │
│ all  │   request    │ quotation  │    invoice     │   receipt    │  return   │
└──────┴──────────────┴────────────┴────────────────┴──────────────┴───────────┘
```

### فلتر كل تبويب

| التبويب | جدول المصدر | فلتر Stage |
|---------|-------------|-----------|
| **الكل** | `purchase_transactions` | كل الـ stages |
| **طلبات الشراء** | `purchase_transactions` | `stage = 'request'` |
| **عروض السعر** | `purchase_transactions` | `stage = 'quotation'` |
| **فواتير مشتريات** | `purchase_transactions` | `stage IN ('draft', 'confirmed', 'received', 'posted')` |
| **اذون تسليم** | `purchase_receipts` | مرتبطة بفواتير مؤكدة |
| **مرتجعات** | `purchase_returns` | مرتبطة بفواتير مستلمة |

### أعمدة الكانبان (6 أعمدة)

```
┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
│  طلبات   │ │  عروض    │ │ فواتير   │ │  اذون    │ │ مرتجعات  │ │  ملغاة   │
│ request  │ │quotation │ │ invoice  │ │ receipt  │ │  return  │ │cancelled │
└──────────┘ └──────────┘ └──────────┘ └──────────┘ └──────────┘ └──────────┘
```

> **ملاحظة**: عمود "فواتير" في الكانبان يشمل: draft, confirmed, received, posted
> كل بطاقة فاتورة تعرض badges للحالات: دفع + تسليم + ترحيل

## ⚠️ قواعد ملزمة

1. **ممنوع إنشاء جداول جديدة** للمراحل — كل شيء في `purchase_transactions` / `sales_transactions`
2. **ممنوع نسخ المستند** عند الانتقال بين المراحل — تغيير `stage` فقط
3. **ممنوع إعطاء رقم نهائي** قبل مرحلة `confirmed`
4. **ممنوع إلغاء ترحيل** مستند `posted`
5. **واجب** عرض أعلام الدفع+التسليم+الترحيل على البطاقة — ليس كتبويبات منفصلة
6. **واجب** دعم السحب والإفلات في الكانبان مع قواعد الانتقال
7. **واجب** تسجيل كل انتقال في `stage_logs` / `activity_log` للتتبع
