# 🔍 رأي نقدي احترافي — هل الجدول الموحد هو الحل الصحيح؟

> **التاريخ**: 2026-02-15  
> **الكاتب**: تحليل هندسي مبني على واقع قاعدة الكود

---

## ⚖️ أولاً: المقارنة المنصفة

### النهج الحالي (جداول متعددة) — كيف تعمل الأنظمة الكبرى

| النظام | الطريقة | لماذا؟ |
|--------|---------|--------|
| **Odoo** | جداول منفصلة (`sale.order`, `account.move`, `stock.picking`) | كل مرحلة لها بنية مختلفة |
| **ERPNext** | DocTypes منفصلة (`Sales Order`, `Sales Invoice`, `Delivery Note`) | كل DocType مستقل |
| **SAP** | جداول منفصلة (`VBAK/VBAP`, `LIKP/LIPS`, `VBRK/VBRP`) | كل مستند له جدول |
| **Oracle EBS** | جداول منفصلة (`OE_ORDER_HEADERS`, `AP_INVOICES`) | نفس المنطق |

**الحقيقة المرة**: لا يوجد نظام ERP عالمي واحد يستخدم جدول واحد لكل شيء.

### لماذا يستخدمون جداول منفصلة؟

1. **كل مرحلة لها حقول مختلفة**: عرض السعر له `valid_until`، الفاتورة لها `due_date`، إذن التسليم له `shipping_method` — وضعها كلها في جدول واحد = عشرات الأعمدة الفارغة (NULL)
2. **الأداء**: جدول واحد مع 11 مرحلة = استعلامات أبطأ مع نمو البيانات
3. **الأمان**: RLS أسهل على جداول منفصلة (مثلاً: فريق المبيعات يرى فقط `sales_invoices`)
4. **التقارير**: تقرير المبيعات يحتاج `WHERE transaction_type = 'sale' AND current_stage = 'invoice'` مقابل `FROM sales_invoices` مباشرة

## ❌ مخاطر الجدول الموحد (ما لا يقوله لك أحد)

### 1. مشكلة "الجدول السمين" (Fat Table Anti-Pattern)

```
transactions:
  id, tenant_id, company_id, transaction_type, current_stage,
  
  -- حقول عرض السعر (فارغة لـ 80% من السجلات)
  quotation_no, quotation_date, quotation_valid_until, quotation_by,
  
  -- حقول الأمر (فارغة لـ 60% من السجلات)
  order_no, order_date, order_approved_by, order_approved_at,
  
  -- حقول التسليم (فارغة لـ 50% من السجلات)
  delivery_no, receipt_no, delivery_date, delivery_by,
  
  -- حقول الفاتورة (فارغة لـ 40% من السجلات)
  invoice_no, invoice_date, invoice_due_date, invoice_by,
  
  -- + 30 عمود آخر...
  
  = إجمالي ~50 عمود، معظمها NULL
```

**النتيجة**: PostgreSQL يخزن NULL بكفاءة (1 bit)، لكن:
- الاستعلامات تصبح معقدة
- الفهارس تصبح أكبر
- الصيانة أصعب

### 2. مشكلة الـ Constraints المتضاربة

```sql
-- كيف تفرض أن الفاتورة يجب أن يكون لها due_date؟
-- في جداول منفصلة: ALTER TABLE invoices ALTER COLUMN due_date SET NOT NULL
-- في الجدول الموحد: مستحيل! لأن المسودة ليس لها due_date
--   → الحل: CHECK constraint معقد:
CHECK (
  (current_stage != 'invoice') OR 
  (current_stage = 'invoice' AND invoice_date IS NOT NULL AND invoice_due_date IS NOT NULL)
)
-- هذا يتكرر لكل مرحلة = عشرات الـ constraints
```

### 3. مشكلة التعقيد الخفي

```
جداول منفصلة:  SELECT * FROM purchase_invoices WHERE supplier_id = $1
جدول موحد:     SELECT * FROM transactions 
               WHERE transaction_type = 'purchase' 
               AND current_stage IN ('invoice', 'paid', 'partial_paid')
               AND party_id = $1
```

**ما يبدو بسيطاً يصبح معقداً في كل مكان.**

### 4. حجم التأثير على الكود الحالي

```
📊 إحصائية الأثر الفعلي:

31 ملف يرجع للجداول القديمة
229 reference في الكود

الأكثر تأثراً:
  purchase_invoices:  49 مرجع في 19 ملف  ←← سيكسر 19 ملف!
  sales_invoices:     28 مرجع في 15 ملف
  purchase_receipts:  27 مرجع في 10 ملف
  purchase_orders:    23 مرجع في 10 ملف
  sales_orders:       21 مرجع في 11 ملف
  quotations:         20 مرجع في 10 ملف
```

**حذف الجداول = كسر 31 ملف و 229+ سطر كود فوراً.**

## 🏆 توصيتي النهائية — نهجان بحسب الأولوية

### النهج A: "الآمن والسريع" — تحسين ما هو موجود ⭐ أنصح به

```
بدلاً من إعادة بناء كل شيء:

1. ✅ أبقِ الجداول الحالية (لا حذف!)
2. ✅ أضف حقل current_stage لكل جدول تجاري موجود
3. ✅ أنشئ جدول واحد bridge: transaction_lifecycle
   → يربط: quotation_id ↔ order_id ↔ invoice_id
   → يتتبع: المرحلة الحالية للدورة
4. ✅ أضف Function: advance_stage() تعمل على الربط
5. ✅ أضف AutoSave + أزرار ديناميكية للـ Frontend
6. ✅ أضف إعدادات الترقيم والموافقات

المزايا:
  → لا يكسر أي كود حالي (0 ملف يتأثر!)
  → يمكن تنفيذه في يوم واحد
  → آمن 100%
  → يحقق كل المتطلبات (AutoSave, أزرار, ترقيم, إعدادات)

العيب:
  → الجداول المتعددة تبقى (لكن مع ربط ذكي)
```

### النهج B: "إعادة البناء الشاملة" — الجدول الموحد

```
1. إنشاء transactions + transaction_items
2. هجرة البيانات
3. تعديل 31 ملف و 229+ سطر
4. اختبار شامل
5. حذف الجداول القديمة

المزايا:
  → هيكل نظيف جداً
  → صيانة أسهل على المدى الطويل

المخاطر:
  → 4-5 أيام عمل
  → احتمال bugs عالي
  → الميزات الشغالة قد تتأثر (POS, receipts, posting)
  → إذا فشل شيء = rollback معقد
```
