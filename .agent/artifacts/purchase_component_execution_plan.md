## ⚖️ سياسات الاستلام المختلف وفروقات الكميات (Receipt Discrepancy Policies)

> **المشكلة**: عند استلام البضائع، قد تختلف الكمية المُستلمة عن الكمية المطلوبة أو المُفوترة.  
> كيف نُعالج هذا الفرق محاسبياً وتسعيرياً؟  
> هذا القسم يوثق كل السيناريوهات العالمية + التوصية لنظام TexaCore.

### 📊 الحالات الممكنة (4 سيناريوهات رئيسية)

```
╔═══════════════════════════════════════════════════════════════════════════════════╗
║ الحالة 1: استلام أقل من المطلوب (Under-Receipt)                                  ║
║   مثال: طلبنا 100 رول ← وصل 80 رول                                              ║
║   السؤال: هل ندفع 100 أم 80؟ وبأي سعر للوحدة؟                                   ║
╠═══════════════════════════════════════════════════════════════════════════════════╣
║ الحالة 2: استلام أكثر من المطلوب (Over-Receipt)                                   ║
║   مثال: طلبنا 100 رول ← وصل 110 رول                                              ║
║   السؤال: هل نقبل الـ 10 الزيادة؟ بأي تكلفة؟                                     ║
╠═══════════════════════════════════════════════════════════════════════════════════╣
║ الحالة 3: فرق في السعر (Price Variance)                                           ║
║   مثال: الأمر بـ 50$/رول ← الفاتورة بـ 52$/رول                                   ║
║   السؤال: هل المخزون بـ 50 أم 52؟ أين يذهب الفرق؟                                ║
╠═══════════════════════════════════════════════════════════════════════════════════╣
║ الحالة 4: فرق في الكمية + السعر معاً (Combined Variance)                          ║
║   مثال: 100 رول × 50$ ← وصل 80 رول × 52$                                        ║
║   السؤال: أعقد حالة — كيف نُوزع التكلفة العادلة؟                                  ║
╚═══════════════════════════════════════════════════════════════════════════════════╝
```

### 🏗️ السيناريوهات العالمية (5 سياسات)

---

#### 🅰️ السياسة 1: الفوترة على المُستلم (Bill Based on Receipt) ← ⭐ الأكثر شيوعاً

> **المبدأ**: ادفع فقط ما استلمت — الفاتورة تتعدل تلقائياً
> **المرجع**: Odoo (Default), ERPNext (Default)
> **مثالي لـ**: الأقمشة، المواد الخام، المواد القابلة للتلف

```
مثال: طلبنا 100 رول × 50$ = 5,000$
       وصل 80 رول

النتيجة:
  الفاتورة تتعدل ← 80 × 50$ = 4,000$
  سعر الوحدة في المخزون = 50$/رول (لم يتغير)
  المتبقي 20 رول: يبقى كـ "بانتظار استلام" أو يُلغى
```

**القيد المحاسبي:**
```
مدين: المخزون (Inventory)           4,000$
دائن: ذمم الموردين (AP)             4,000$

(لا يوجد فرق — لأن الفاتورة = المُستلم)
```

**المميزات**: بسيط + عادل + لا فروقات
**العيوب**: يحتاج آلية لإغلاق الكمية المتبقية

---

#### 🅱️ السياسة 2: إعادة توزيع التكلفة (Cost Redistribution)

> **المبدأ**: الفاتورة تبقى كما هي — سعر الوحدة يرتفع
> **المرجع**: شائع في صناعة الأقمشة والاستيراد
> **مثالي لـ**: عندما المورد لا يُصدر إشعار دائن ويرفض تعديل الفاتورة

```
مثال: طلبنا 100 رول × 50$ = 5,000$ (فاتورة المورد)
       وصل 80 رول (المورد يرفض التعديل)

النتيجة:
  الفاتورة تبقى = 5,000$
  سعر الوحدة الجديد = 5,000$ ÷ 80 = 62.50$/رول ← ⚠️ أعلى!
  رأس المال (تكلفة المادة) = 62.50$ بدل 50$
```

**القيد المحاسبي:**
```
مدين: المخزون (Inventory)           5,000$  (80 رول × 62.50$)
دائن: ذمم الموردين (AP)             5,000$

(كل المبلغ يدخل المخزون — سعر الوحدة أعلى)
```

**المميزات**: يعكس التكلفة الحقيقية — المورد لم يُنقص الفاتورة
**العيوب**: سعر الوحدة ليس الأصلي — يؤثر على هامش الربح

---

#### 🅲️ السياسة 3: إشعار بالفرق (Debit Note / Credit Note)

> **المبدأ**: الفاتورة تبقى + إنشاء إشعار مدين بالفرق → يُخصم من حساب المورد
> **المرجع**: SAP (Standard), Oracle
> **مثالي لـ**: العلاقات الرسمية مع الموردين + عقود واضحة

```
مثال: طلبنا 100 رول × 50$ = 5,000$ (فاتورة أصلية)
       وصل 80 رول

النتيجة:
  الفاتورة الأصلية = 5,000$ (تبقى)
  إشعار مدين (Debit Note) = 20 × 50$ = 1,000$
  صافي المستحق للمورد = 5,000$ - 1,000$ = 4,000$
  سعر الوحدة في المخزون = 50$/رول (لم يتغير)
```

**القيود المحاسبية:**
```
عند الاستلام:
  مدين: المخزون (Inventory)           4,000$  (80 × 50$)
  مدين: بضاعة بالطريق أو استلام مؤقت   1,000$  (20 × 50$ — مؤقت)
  دائن: ذمم الموردين (AP)             5,000$

عند إصدار إشعار مدين:
  مدين: ذمم الموردين (AP)             1,000$
  دائن: بضاعة بالطريق                 1,000$  (إغلاق المؤقت)
```

**المميزات**: محاسبة نظيفة + سجل واضح + المورد يعرف
**العيوب**: خطوات أكثر + يحتاج موافقة المورد أحياناً

---

#### 🅳️ السياسة 4: حساب فروقات (Variance Account)

> **المبدأ**: الفاتورة تبقى + الفرق يذهب لحساب "فروقات المشتريات" تلقائياً
> **المرجع**: SAP (Standard Cost), Oracle (Standard Costing)
> **مثالي لـ**: الشركات الكبيرة + التكلفة المعيارية + التقارير الإدارية

```
مثال: طلبنا 100 رول × 50$ = 5,000$
       وصل 80 رول

النتيجة:
  الفاتورة = 5,000$ (لا تتغير)
  المخزون = 80 × 50$ = 4,000$ (بالسعر المعياري)
  حساب فروقات المشتريات = 1,000$ (الفرق)
```

**القيد المحاسبي:**
```
مدين: المخزون (Inventory)           4,000$  (80 × 50$)
مدين: فروقات مشتريات (PPV)          1,000$  (الفرق)
دائن: ذمم الموردين (AP)             5,000$

(الفرق يظهر كمصروف في فروقات المشتريات — لا يؤثر على سعر المادة)
```

**المميزات**: سعر الوحدة ثابت + الفروقات ظاهرة في التقارير + نظيف
**العيوب**: يحتاج إعداد حساب فروقات + قد يخفي مشاكل أسعار حقيقية

---

#### 🅴️ السياسة 5: حدود تَسامُح (Tolerance) + قبول تلقائي

> **المبدأ**: إذا الفرق ضمن نسبة معينة (مثلاً ±5%) → يُقبل تلقائياً
> **المرجع**: SAP (Tolerance Limits), Oracle (Receiving Tolerances)
> **مثالي لـ**: عمليات ذات حجم كبير + تقليل العمل اليدوي

```
مثال: Tolerance = 5%
       طلبنا 100 رول ← وصل 97 رول (فرق 3%)

النتيجة:
  3% < 5% → ✅ قبول تلقائي
  الفاتورة تتعدل ← 97 × 50$ = 4,850$
  لا تنبيه ولا إشعار

مثال 2: وصل 90 رول (فرق 10%)
  10% > 5% → ⚠️ يحتاج موافقة يدوية
  → يُفعَّل إحدى السياسات أعلاه (A/B/C/D)
```

### 💰 تأثير كل سياسة على تكلفة المادة (Inventory Valuation)

```
╔═══════════════════════════════════╦══════════════════╦══════════════════════════╗
║ السياسة                           ║ سعر الوحدة       ║ أثر على رأس المال        ║
╠═══════════════════════════════════╬══════════════════╬══════════════════════════╣
║ 🅰 Bill on Receipt                ║ 50$ (الأصلي)     ║ ✅ لا تغيير              ║
║ 🅱 Cost Redistribution            ║ 62.50$ (أعلى!)   ║ ⚠️ يرفع التكلفة         ║
║ 🅲 Debit Note                     ║ 50$ (الأصلي)     ║ ✅ لا تغيير              ║
║ 🅳 Variance Account               ║ 50$ (الأصلي)     ║ ✅ لا تغيير + فرق بحساب   ║
║ 🅴 Tolerance + Auto               ║ 50$ (الأصلي)     ║ ✅ لا تغيير (ضمن الحد)    ║
╚═══════════════════════════════════╩══════════════════╩══════════════════════════╝
```

### 💵 سياسات فرق السعر (عندما الكمية مطابقة لكن السعر مختلف)

```
╔══════════════════════════════════════════════════════════════════════════════════╗
║ مثال: أمر شراء 100 × 50$ = 5,000$                                              ║
║       فاتورة المورد: 100 × 52$ = 5,200$ (فرق +200$)                            ║
╠══════════════════════════════════════════════════════════════════════════════════╣
║                                                                                  ║
║ الطريقة 1: تطبيق سعر الفاتورة (Invoice Price Takes Precedence)                  ║
║   → المخزون = 100 × 52$ = 5,200$                                                ║
║   → سعر الوحدة الجديد = 52$                                                     ║
║   → مثالي لـ: متوسط التكلفة المتحرك (Moving Average)                             ║
║                                                                                  ║
║ الطريقة 2: تثبيت سعر الأمر + فرق لحساب فروقات                                   ║
║   → المخزون = 100 × 50$ = 5,000$                                                ║
║   → فروقات مشتريات = 200$                                                        ║
║   → مثالي لـ: التكلفة المعيارية (Standard Cost)                                  ║
║                                                                                  ║
║ الطريقة 3: متوسط مرجح                                                           ║
║   → إذا كان المخزون الحالي = 50 رول × 48$                                       ║
║   → المتوسط الجديد = (50×48 + 100×52) ÷ 150 = 50.67$                            ║
║                                                                                  ║
╚══════════════════════════════════════════════════════════════════════════════════╝
```

### 🔄 حساب GR/IR (Goods Receipt / Invoice Receipt) — نمط SAP

> **GR/IR** هو حساب وسيط يُستخدم عندما يكون الاستلام والفوترة عمليتين منفصلتين.

```
الاستلام أولاً ← الفاتورة لاحقاً:
══════════════════════════════════

1️⃣ عند استلام البضائع (قبل وصول الفاتورة):
   مدين: المخزون (بسعر أمر الشراء)      4,000$
   دائن: بضاعة مُستلمة لم تُفوتر (GR/IR) 4,000$

2️⃣ عند وصول فاتورة المورد:
   مدين: بضاعة مُستلمة لم تُفوتر (GR/IR) 4,000$
   دائن: ذمم الموردين (AP)               4,000$

   → الحساب الوسيط يصبح = 0 ← ✅ مُقفل


الفاتورة أولاً ← الاستلام لاحقاً:
══════════════════════════════════

1️⃣ عند وصول الفاتورة (قبل الاستلام):
   مدين: بضاعة بالطريق (GIT)            5,000$
   دائن: ذمم الموردين (AP)              5,000$

2️⃣ عند استلام 80 من 100:
   مدين: المخزون                         4,000$  (80 × 50$)
   دائن: بضاعة بالطريق (GIT)            4,000$
   
   → بضاعة بالطريق المتبقي = 1,000$ (20 رول لم تصل بعد)
```

### 🔺 المطابقة الثلاثية (3-Way Matching)

```
╔══════════════════════════════════════════════════════════════════╗
║  PO (أمر الشراء)   ↔   GRN (إذن الاستلام)   ↔   Invoice (الفاتورة)  ║
║  100 × 50$ = 5,000$    80 × 50$ = 4,000$      100 × 52$ = 5,200$ ║
║                                                                    ║
║  فرق كمية: 100 vs 80 = ⚠️ -20 رول                                 ║
║  فرق سعر: 50$ vs 52$ = ⚠️ +2$/وحدة                                ║
║  فرق إجمالي: 5,000$ vs 4,000$ vs 5,200$                          ║
║                                                                    ║
║  النظام يُنبّه:                                                    ║
║    "⚠️ فرق كمية: الأمر 100، المُستلم 80 (-20%)"                   ║
║    "⚠️ فرق سعر: الأمر 50$، الفاتورة 52$ (+4%)"                   ║
║    "📋 اختر سياسة المعالجة: [أ] [ب] [ج] [د]"                     ║
╚══════════════════════════════════════════════════════════════════╝
```

#### الكود المقترح (ReceiptDiscrepancyHandler):
```typescript
interface ReceiptDiscrepancy {
  itemId: string;
  materialName: string;
  orderedQty: number;
  receivedQty: number;
  orderPrice: number;       // سعر أمر الشراء
  invoicePrice: number;     // سعر الفاتورة (قد يختلف)
  qtyDiffPercent: number;   // نسبة فرق الكمية
  priceDiffPercent: number; // نسبة فرق السعر
}

interface DiscrepancyResolution {
  policy: 'bill_on_receipt' | 'cost_redistribution' | 'debit_note' | 'variance_account';
  adjustedInvoiceTotal: number;
  unitCostForInventory: number;
  varianceAmount: number;
  varianceAccountId?: string;
  debitNoteAmount?: number;
  journalEntries: JournalEntryLine[];
}

async function resolveReceiptDiscrepancy(
  discrepancy: ReceiptDiscrepancy,
  settings: CompanyAccountingSettings
): Promise<DiscrepancyResolution> {
  
  const { receipt_variance_policy, receipt_qty_tolerance_percent } = settings;
  const withinTolerance = Math.abs(discrepancy.qtyDiffPercent) <= receipt_qty_tolerance_percent;
  
  // ضمن الحد → قبول تلقائي بسياسة Bill on Receipt
  if (withinTolerance) {
    return billOnReceipt(discrepancy);
  }
  
  // خارج الحد → تطبيق السياسة المُعدّة
  switch (receipt_variance_policy) {
    case 'bill_on_receipt':
      return billOnReceipt(discrepancy);
    case 'cost_redistribution':
      return costRedistribution(discrepancy);
    case 'debit_note':
      return createDebitNote(discrepancy);
    case 'variance_account':
      return postToVarianceAccount(discrepancy, settings);
    case 'manual':
      throw new ManualDecisionRequired(discrepancy); // يعرض حوار للمستخدم
  }
}
```

## 📒 استراتيجية القيود المحاسبية عند الاختلاف

> **🔒 القاعدة الذهبية**: القيد المُرحّل لا يُعدّل أبداً — يُنشأ **قيد تسوية/تعديل** منفصل  
> هذا هو المعيار العالمي في SAP, Oracle, Odoo وكل الأنظمة المحاسبية المحترمة.

### لماذا لا نعدل القيد الأصلي؟
```
╔══════════════════════════════════════════════════════════════════╗
║ 1. 📜 المسار المحاسبي (Audit Trail)                             ║
║    → كل تغيير يجب أن يكون مُوثّقاً كعملية منفصلة               ║
║    → المدققون يحتاجون رؤية القيد الأصلي + التعديل              ║
║                                                                  ║
║ 2. ⚖️ معايير المحاسبة (IFRS/GAAP)                               ║
║    → القيد المُرحّل = حقيقة محاسبية مُوثّقة                     ║
║    → التعديل يتم بقيد عكسي + قيد صحيح = أثر نفسه لكن بسجل      ║
║                                                                  ║
║ 3. 🔐 الأمان                                                     ║
║    → منع التلاعب بالقيود القديمة                                 ║
║    → القيد المُرحّل = مُقفل (locked)                             ║
║                                                                  ║
║ 4. 📊 التقارير                                                   ║
║    → تقارير الفترة تبقى صحيحة                                    ║
║    → قيد التسوية يظهر في الفترة التي حصل فيها الاختلاف           ║
╚══════════════════════════════════════════════════════════════════╝
```

### أنواع القيود المحاسبية في دورة المشتريات:

```
╔════════════════╦════════════════════════╦═══════════════════════════════╗
║ نوع القيد       ║ متى يُنشأ؟              ║ هل يُعدّل؟                     ║
╠════════════════╬════════════════════════╬═══════════════════════════════╣
║ 1. قيد الشراء  ║ عند ترحيل الفاتورة     ║ ❌ لا يُعدّل أبداً            ║
║    (أصلي)       ║ stage → posted          ║ مقفل نهائياً                  ║
╠════════════════╬════════════════════════╬═══════════════════════════════╣
║ 2. قيد تسوية   ║ عند اكتشاف فرق        ║ ❌ قيد منفصل برقم جديد        ║
║    (تعديل)      ║ بعد الاستلام/المطابقة  ║ يرجع (references) للقيد الأصلي║
╠════════════════╬════════════════════════╬═══════════════════════════════╣
║ 3. قيد عكسي    ║ عند إلغاء ترحيل        ║ ❌ قيد منفصل يعكس الأصلي     ║
║    (إلغاء)      ║ أو إنشاء مرتجع        ║ نفس المبالغ بعكس الأطراف      ║
╠════════════════╬════════════════════════╬═══════════════════════════════╣
║ 4. قيد دفع     ║ عند تسجيل دفعة        ║ ❌ قيد منفصل                  ║
║                 ║                        ║ يربط بالفاتورة                ║
╚════════════════╩════════════════════════╩═══════════════════════════════╝
```

### مثال تفصيلي — قيد تسوية عند اختلاف الكمية:

```
══════════════════════════════════════════════════════════════════════
السيناريو: فاتورة 100 رول × 50$ = 5,000$ (مُرحّلة)
           وصل 80 رول فقط
           سياسة الشركة: Variance Account (حساب فروقات)
══════════════════════════════════════════════════════════════════════

🔒 القيد الأصلي (JE-2026-0150) — عند الترحيل — لا يتغير:
┌────────────────────────────────────────────────────────────┐
│  التاريخ: 2026-02-10                                       │
│  المرجع: فاتورة مشتريات PI-2026-045                        │
│  ─────────────────────────────────────────────────────      │
│  مدين: المخزون (بضائع)              5,000$                 │
│  دائن: ذمم الموردين — شركة X        5,000$                 │
│  ─────────────────────────────────────────────────────      │
│  الحالة: ✅ مُرحّل — 🔒 مقفل                                │
└────────────────────────────────────────────────────────────┘

📝 قيد التسوية (JE-ADJ-2026-0012) — عند تأكيد الاستلام — قيد جديد:
┌────────────────────────────────────────────────────────────┐
│  التاريخ: 2026-02-15                                       │
│  المرجع: تسوية استلام — فاتورة PI-2026-045                 │
│  السبب: فرق كمية — مستلم 80 من 100 (-20 رول)              │
│  القيد المرجعي: JE-2026-0150                               │
│  ─────────────────────────────────────────────────────      │
│  مدين: فروقات مشتريات (PPV)         1,000$  (20 × 50$)    │
│  دائن: المخزون (بضائع)              1,000$                 │
│  ─────────────────────────────────────────────────────      │
│  الحالة: ✅ مُرحّل تلقائياً                                 │
│  النوع: adjustment (تسوية)                                  │
└────────────────────────────────────────────────────────────┘

📊 النتيجة النهائية (صافي القيدين):
┌────────────────────────────────────────────────────────────┐
│  المخزون: 5,000 - 1,000 = 4,000$ ← (80 رول × 50$) ✅     │
│  ذمم الموردين: 5,000$ ← (المبلغ المستحق) ✅               │
│  فروقات مشتريات: 1,000$ ← (مصروف الفرق) ✅               │
└────────────────────────────────────────────────────────────┘
```

### قيود التسوية لكل سياسة:

| السياسة | القيد الأصلي | قيد التسوية | النتيجة الصافية |
|---------|-------------|-------------|----------------|
| 🅰 Bill on Receipt | ❌ لا يُنشأ قيد حتى تأكيد الاستلام | القيد الأول = على الكمية المُستلمة | مخزون = 4,000$ · AP = 4,000$ |
| 🅱 Cost Redistribution | مخزون 5,000 / AP 5,000 | ❌ لا يحتاج تسوية (الفاتورة = المدفوع) | مخزون = 5,000$ · 80 رول × 62.50$ |
| 🅲 Debit Note | مخزون 5,000 / AP 5,000 | مدين: AP 1,000 / دائن: مخزون 1,000 + إشعار مدين | مخزون = 4,000$ · AP = 4,000$ |
| 🅳 Variance Account | مخزون 5,000 / AP 5,000 | مدين: PPV 1,000 / دائن: مخزون 1,000 | مخزون = 4,000$ · AP = 5,000$ · PPV = 1,000$ |

### حقول القيد الجديدة في DB:
```sql
-- إضافة لجدول journal_entries أو accounting_entries:
ALTER TABLE journal_entries ADD COLUMN IF NOT EXISTS
    entry_type TEXT DEFAULT 'standard'
        CHECK (entry_type IN (
            'standard',       -- قيد عادي (شراء، بيع، مصروف...)
            'adjustment',     -- قيد تسوية (فرق استلام، فرق سعر...)
            'reversal',       -- قيد عكسي (إلغاء ترحيل)
            'payment',        -- قيد دفع
            'closing'         -- قيد إقفال
        )),
    parent_entry_id UUID REFERENCES journal_entries(id),  -- القيد الأصلي المرجعي
    adjustment_reason TEXT;  -- سبب التسوية (للتقارير)
```

## 📌 قواعد العمل أثناء التنفيذ

1. **الجدول الموحد هو المصدر**: كل شيء يمر عبر `purchase_transactions` — لا نُنشئ جداول جديدة للمستندات
2. **Stage بدل DocType**: الفلترة بـ `WHERE stage = 'order'` بدل جداول منفصلة
3. **Config-driven**: كل مرحلة مُعرّفة في `stageConfig.ts` — لا hardcoding
4. **RBAC أولاً**: كل ميزة تُخفى/تُظهر حسب الصلاحية قبل التنفيذ
5. **AutoSave = draft فوري**: فتح مكون جديد = INSERT بحالة draft + AutoSave
6. **الترقيم عند التأكيد**: المسودة بدون رقم مرئي — أول تأكيد = أول رقم (PQ-xxx)
7. **Edit Mode Pattern**: مستند مؤكد = مقفل → زر تعديل → فتح → تأكيد/إلغاء
8. **القيد = مسودة حتى الترحيل**: يظهر في تبويب المالية كقراءة فقط
9. **🔒 القيد المُرحّل لا يُعدّل**: عند الاختلاف → قيد تسوية منفصل (لا تعديل على الأصلي)
10. **المصاريف من الصندوق**: لا تُدخل يدوياً في الفاتورة — تربط من treasury
11. **الإعدادات تُحترم**: كل سياسة (فوترة، تقييم، تسامح) تُقرأ من `company_accounting_settings`
12. **تحديث هذا الملف أولاً**: عند إكمال أي مهمة، حدّث ✅ هنا
