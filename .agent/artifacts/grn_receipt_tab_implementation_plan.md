## GRN Receipt Tab & Accounting Integration Plan

## 📊 تحليل الوضع الحالي

### المشاكل المُكتشفة:
| # | المشكلة | الموقع | التأثير |
|---|---------|--------|---------|
| 1 | أمر الشراء يظهر "received" لكن بالكميات الأصلية (1,300) وليست الفعلية (2,502) | `PurchaseCycleList` + `UnifiedTradeSheet` | المستخدم يرى بيانات خاطئة |
| 2 | لا يوجد تبويب استلام في أمر الشراء أو فاتورة المشتريات | `tradeConfigs.ts` | لا يمكن رؤية تفاصيل الاستلام |
| 3 | القيد المحاسبي يُنشأ بدون `account_id` حقيقي | `receiptCompletionService.ts` (سطر 494) | القيد غير مرتبط بشجرة الحسابات |
| 4 | القيد لا يظهر عند فتح أمر الشراء/الفاتورة | لا يوجد tab لعرضه | المحاسب لا يرى القيد |
| 5 | الكمية في `purchase_receipt_items` لا تربط بالـ source item | `createReceiptItems` | لا مقارنة بين المطلوب والمستلم |

### البنية الحالية:
```
purchase_orders → [status: received]
     ↓
purchase_receipts → [status: completed, order_id: ...]
     ↓
purchase_receipt_items → [quantity_received: X, product_id: ...]
     ↓
inventory_movements → [quantity: X, status: completed]
     ↓
journal_entries → [بدون account_id حقيقي!]
```

### التبويبات الحالية:
- **أمر الشراء**: الأصناف ← متصفح المواد ← المورد ← الشحن ← المرفقات ← النشاط
- **فاتورة المشتريات**: الأصناف ← متصفح المواد ← المورد ← المدفوعات ← المرفقات ← النشاط
- **إذن الاستلام (GRN)**: الأصناف ← متصفح المواد ← استلام المستودع ← المرفقات ← النشاط

## 🎯 الخطة المرحلية

### المرحلة 1️⃣: إنشاء تبويب "ملخص الاستلام" (ReceiptSummaryTab)
**الهدف**: تبويب جديد يظهر في أمر الشراء وفاتورة المشتريات يعرض تفاصيل الاستلام

**الملفات المطلوبة**:
1. `src/features/accounting/components/unified/tabs/ReceiptSummaryTab.tsx` — **جديد**
2. `src/features/accounting/components/unified/configs/tradeConfigs.ts` — إضافة التبويب

**المحتوى**:
```
┌─────────────────────────────────────────────────┐
│                 ملخص الاستلام                     │
├─────────────────────────────────────────────────┤
│ 📦 رقم إذن الاستلام: GRN-20260213-HE19         │
│ 📅 تاريخ الاستلام: 2026/02/13                   │
│ 🏭 المستودع: مستودع 4 أوديسا                     │
│ 📊 الحالة: مكتمل ✅                              │
├─────────────────────────────────────────────────┤
│                                                   │
│  المادة     | المطلوب | المستلم | الفرق | الحالة  │
│  ─────────────────────────────────────────────── │
│  كتان تركي  | 1,300  | 2,502  | +1,202 | زيادة   │
│                                                   │
├─────────────────────────────────────────────────┤
│ 🔗 القيد المحاسبي: JE-GRN-20260213-HE19        │
│    مدين: مخزون البضاعة ← 5,204 USD              │
│    دائن: بضاعة غير مفوترة ← 5,204 USD           │
│    (يظهر للمدير فقط)                             │
└─────────────────────────────────────────────────┘
```

**خطوات التنفيذ**:
1. إنشاء `ReceiptSummaryTab.tsx`:
   - يستقبل `sourceDocumentId` (PO ID أو Invoice ID)
   - يبحث في `purchase_receipts` عن أي receipt مرتبط بهذا المستند
   - يجلب `purchase_receipt_items` للمقارنة
   - يجلب القيد المحاسبي (`journal_entries`) المرتبط بالـ receipt
   - يعرض جدول مقارنة: المطلوب vs المستلم vs الفرق
   - القيد المحاسبي يظهر فقط للمدير (RBAC check)

2. إضافة TAB في `tradeConfigs.ts`:
   ```typescript
   receiptSummary: {
       id: 'receipt_summary',
       labelKey: 'trade.tabs.receiptSummary',
       icon: 'ClipboardCheck',
       component: 'ReceiptSummaryTab',
   },
   ```

3. إضافة التبويب لـ `purchaseOrderConfig.tabs` و `purchaseInvoiceConfig.tabs`

4. Handle tab rendering في `UnifiedAccountingSheet.tsx` (case 'receipt_summary')

### المرحلة 2️⃣: إصلاح القيد المحاسبي (account_id حقيقي)
**الهدف**: ربط القيد بحسابات حقيقية من شجرة الحسابات

**الملفات المتأثرة**:
1. `receiptCompletionService.ts` — `handleAccountingEntry()`

**المنطق**:
```
عند الاستلام من أمر شراء:
  مدين: حـ/ المخزون (بالكمية الفعلية × سعر الوحدة)
  دائن: حـ/ بضاعة مستلمة غير مفوترة (GRNI)

عند الاستلام من فاتورة مشتريات:
  مدين: حـ/ المخزون (بالكمية الفعلية × سعر الوحدة)
  دائن: حـ/ الموردون (بقيمة الفاتورة)
  + إذا كان هناك فرق:
    مدين/دائن: حـ/ فروقات الاستلام
```

**خطوات التنفيذ**:
1. البحث عن الحسابات بالـ `account_code` أو `account_name` من `chart_of_accounts`
2. إدراج `account_id` الفعلي في `journal_entry_lines`
3. إضافة validation: إذا لم يوجد الحساب → إنشاؤه تلقائياً أو تحذير

### المرحلة 3️⃣: تحديث الكميات في المستودع

**ما يحتاج إصلاح**: ربط `inventory_movements.material_id` بالمادة الصحيحة بدلاً من UUID group
