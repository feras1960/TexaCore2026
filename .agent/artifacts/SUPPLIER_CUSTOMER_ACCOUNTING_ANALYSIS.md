# 📊 تحليل شامل: نظام حسابات الموردين والعملاء

## 📅 التاريخ: 19 فبراير 2026

---

## 🔍 المشكلة المكتشفة

عند ترحيل فاتورة مشتريات لمورد معين (مثلاً "مصنع النسيج الأوكراني"):
- القيد المحاسبي يسجل الدائن في حساب **عام واحد**: `2111 - دين الموردين - الأقمشة`
- عند فتح صفحة المورد → الرصيد يظهر **صفر**
- **السبب**: لا يوجد ربط مباشر بين حركات الحساب العام والمورد المحدد

---

## 🏗️ الحالة الحالية للنظام

### شجرة الحسابات (حسابات الموردين):
```
211 — دين الموردين (مجموعة) [is_group = true]
  ├── 2111 — دين الموردين - الأقمشة [is_group = false] ← حساب واحد لكل الموردين!
  └── 2112 — دين الموردين - أخرى [is_group = false]
```

### جدول `suppliers`:
- يوجد عمود `payable_account_id` (uuid) — لربط المورد بحساب
- **المشكلة**: أغلب الموردين مربوطين بحساب `الأصول` (!) أو بحساب `2111` العام
- لا يوجد حسابات **فرعية فردية** لكل مورد

### جدول `journal_entry_lines`:
- يوجد عمودين مهمين: `party_type` + `party_id`
- **بعض** القيود تحتوي على `party_type = 'supplier'` و `party_id = UUID المورد`
- **لكن** القيود المنشأة من `purchaseAccountingService` **لا تملأ** هذه الحقول!

### جدول `customers`:
- يوجد عمود `receivable_account_id` (uuid) — نفس المفهوم للعملاء
- نفس المشكلة بالضبط

## ⚖️ المقارنة: النظامان المتاحان

### 🅰️ النظام الحالي: حساب مجمع + Party Tracking (Sub-Ledger)

```
القيد:
  مدين: تكلفة الأقمشة المباعة     9,000
  مدين: ضريبة القيمة المضافة        450
  دائن: دين الموردين - الأقمشة     9,450  ← حساب واحد لكل الموردين
         (party_type: supplier, party_id: UUID_المورد)  ← التتبع الفرعي
```

**المميزات:**
- ✅ شجرة حسابات بسيطة ونظيفة (لا تتضخم)
- ✅ الميزانية العمومية تظهر إجمالي الموردين في سطر واحد
- ✅ مرونة عالية — يمكن إضافة موردين بلا حد
- ✅ **هو المعيار في Odoo, ERPNext, SAP**

**العيوب:**
- ❌ تحتاج برمجة إضافية لـ Sub-Ledger (كشف حساب المورد)
- ❌ رصيد المورد لا يظهر مباشرة من chart_of_accounts — يحتاج تجميع من journal_entry_lines

---

### 🅱️ النظام البديل: حساب فرعي لكل مورد/عميل

```
211 — دين الموردين (مجموعة)
  ├── 211001 — مصنع النسيج الأوكراني
  ├── 211002 — مصانع بورصة للحرير
  ├── 211003 — مصانع مومباي للكتان
  ├── 211004 — مصانع قوانغدونغ
  └── ...
```

**المميزات:**
- ✅ رصيد كل مورد يظهر مباشرة من الشجرة
- ✅ كشف حساب المورد = ببساطة ledger entries للحساب الفرعي
- ✅ **أبسط في البرمجة**

**العيوب:**
- ❌ شجرة الحسابات تتضخم بسرعة (100 مورد = 100 حساب!)
- ❌ عند إضافة مورد جديد → يجب إنشاء حساب محاسبي أيضاً
- ❌ صعوبة في الصيانة والتنظيم
- ❌ **ليس المعيار العالمي — لكنه شائع في الأنظمة الصغيرة**

## 🏆 التوصية: النظام 🅰️ (Sub-Ledger) — مع إصلاح الكود الحالي

### لماذا؟
1. **البنية التحتية موجودة أصلاً!** — عمودي `party_type` و `party_id` في `journal_entry_lines` جاهزين
2. **أكثر احترافية** — هذا ما تستخدمه Odoo, ERPNext, SAP, Oracle
3. **قابل للتوسع** — 1000 مورد بدون تضخم الشجرة
4. **الإصلاح المطلوب بسيط** — فقط 3 نقاط:

### 📋 خطة الإصلاح (3 تعديلات فقط):

#### 1️⃣ إصلاح `purchaseAccountingService.ts` — تعبئة party_type + party_id

```typescript
// السطر الدائن (الموردين)
lines.push({
    account_id: creditAccountId!,
    debit: 0,
    credit: Math.round(effectiveTotal * 100) / 100,
    description: `فاتورة مشتريات #${invoiceRef} - ${supplierName}`,
    cost_center_id: null,
    // ⬇️ الإضافة المطلوبة:
    party_type: 'supplier',
    party_id: invoice.supplier_id,
});
```

#### 2️⃣ إصلاح `journalEntriesService.ts` — دعم party_type + party_id عند الإنشاء

التأكد من أن service الإنشاء يمرر `party_type` و `party_id` للقاعدة.

#### 3️⃣ حساب رصيد المورد — استعلام Sub-Ledger

```sql
-- رصيد مورد محدد:
SELECT 
    SUM(credit) - SUM(debit) as balance
FROM journal_entry_lines jel
JOIN journal_entries je ON jel.entry_id = je.id
WHERE jel.party_type = 'supplier' 
  AND jel.party_id = 'UUID_المورد'
  AND je.status = 'posted'
  AND je.company_id = 'UUID_الشركة';
```

#### 4️⃣ (نفس الشيء للعملاء)

```typescript
// السطر المدين (العملاء)
lines.push({
    account_id: receivableAccountId,
    debit: totalAmount,
    credit: 0,
    party_type: 'customer',
    party_id: invoice.customer_id,
});
```

## 🔄 نفس الإصلاح للعملاء

| البند | الموردين | العملاء |
|-------|---------|---------|
| حساب مجمع | دين الموردين (211) | ذمم العملاء (121) |
| party_type | `supplier` | `customer` |
| عمود الحساب | `payable_account_id` | `receivable_account_id` |
| الاتجاه | دائن ↑ = زيادة دين | مدين ↑ = زيادة مستحق |

## ✅ الملخص

| الجانب | حساب فرعي لكل طرف | Sub-Ledger (party tracking) |
|--------|-------------------|---------------------------|
| تضخم الشجرة | ❌ نعم | ✅ لا |
| سهولة التنفيذ | ✅ أبسط | ⚠️ يحتاج برمجة |
| المعيار العالمي | ❌ لا | ✅ نعم (Odoo/SAP/ERPNext) |
| قابلية التوسع | ❌ محدودة | ✅ عالية |
| **البنية موجودة أصلاً** | — | ✅ party_type + party_id جاهزين |

**🎯 القرار الموصى: استخدام Sub-Ledger (party tracking) وإصلاح الكود ليملأ party_type + party_id**
