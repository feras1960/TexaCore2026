# تحليل دورة المستند الموحد — Unified Document Lifecycle Analysis

## 📌 ملخص طلب المستخدم

المطلوب هو **مستند واحد (record واحد)** يتحرك بين المراحل (طلب → عرض سعر → أمر → فاتورة) بدون نسخ أو إنشاء مستند جديد. الحالة تتغير داخل نفس الشيت/البوب أب، مع:
- لا أثر محاسبي إلا عند مرحلة "الفاتورة"
- سهولة التبديل بين المراحل عبر StatusDropdown
- الإبقاء على التبويبات (tabs) في صفحة القائمة لعرض المستندات حسب المرحلة

---

## 🔍 كيف يعمل Odoo و ERPNext

### Odoo (الأقرب لما نريد)

```
نموذج واحد: purchase.order
─────────────────────────────────────────────────
RFQ (مسودة)  →  RFQ Sent  →  Purchase Order  →  Done
─────────────────────────────────────────────────
     ↓ نفس الـ record — لا ينسخ ↓
```

**كيف يعمل:**
1. **Model واحد** `purchase.order` يُستخدم لكل من RFQ و PO
2. **Status Bar** في أعلى الفورم يُظهر المراحل كشريط أفقي (Draft → Sent → PO → Done)
3. **الضغط على "Confirm Order"** يحوّل RFQ إلى PO — **نفس الـ record**، لا نسخ
4. **الفاتورة (Vendor Bill)** هي مستند **منفصل** (`account.move`) — تُنشأ من الـ PO بزر "Create Bill"
5. **Bill Control Policy**: يمكن إنشاء الفاتورة عند التأكيد أو عند الاستلام

**النقطة المهمة في أودو:**
- الـ RFQ والـ PO هما **نفس المستند** — فقط الحالة تتغير
- لكن الـ **Vendor Bill (الفاتورة)** هي مستند **محاسبي منفصل** لأنها تُنشئ قيود محاسبية

### ERPNext (مستندات منفصلة مرتبطة)

```
Material Request → Supplier Quotation → Purchase Order → Purchase Receipt → Purchase Invoice
      ↓                    ↓                  ↓                ↓                 ↓
 DocType مختلف        DocType مختلف      DocType مختلف    DocType مختلف      DocType مختلف
```

**كيف يعمل:**
1. كل مرحلة = **DocType مختلف** (مستند منفصل في جدول مختلف)
2. يرتبطون عبر **Link Fields** (حقول ربط)
3. كل DocType له حالاته الخاصة (Draft → Submitted → Cancelled)
4. نظام **Amend**: لتعديل مستند مُرسل، يُلغى ويُنشأ نسخة معدّلة

## 📋 خطة التنفيذ المفصّلة

### المرحلة 1: الشيت الموحد (UnifiedTradeSheet)

**الوضع الحالي:**
- `UnifiedTradeSheet` يمرر `type="invoice"` ثابتاً
- الشيت يعرض "فاتورة شراء" دائماً بغض النظر عن المرحلة
- عند تبديل النوع (في المبيعات) كان يحاول نسخ المستند

**المطلوب:**
1. ❌ إزالة مفهوم "النوع" (type) كتحديد للجدول — لأن الجدول واحد
2. ✅ إضافة **Status Bar** أفقي في أعلى الشيت يُظهر المراحل
3. ✅ العنوان يتغير ديناميكياً حسب `document_stage`:
   - `request` → "طلب شراء" / "طلب بيع"
   - `quotation` → "عرض سعر شراء" / "عرض سعر مبيعات"
   - `order` → "أمر شراء" / "أمر بيع"
   - `invoice` → "فاتورة شراء" / "فاتورة مبيعات"
4. ✅ زر **"تأكيد"** يتغير نصه حسب الانتقال المتاح التالي
5. ✅ الحفظ يحفظ **نفس الـ record** — لا ينشئ مستند جديد

### المرحلة 2: Status Bar (شريط الحالة)

**تصميم Odoo-like:**
```
 ● مسودة ──── ○ طلب ──── ○ عرض سعر ──── ○ أمر ──── ○ فاتورة
   ↑ الحالة الحالية
```

- الخطوة النشطة تكون ملونة ومعبّأة
- الخطوات المكتملة تكون بلون باهت مع ✓
- الخطوات المستقبلية تكون بلون رمادي
- يمكن الضغط على خطوة للانتقال إليها (إن كان الانتقال مسموح)

### المرحلة 3: أزرار الإجراء الديناميكية

| المرحلة الحالية | الزر الرئيسي | الأزرار الثانوية |
|---|---|---|
| مسودة (draft) | تأكيد كطلب شراء | إلغاء |
| طلب (requested) | تأكيد كعرض سعر / تأكيد كأمر | رفض، إلغاء |
| عرض سعر (quoted) | تأكيد كأمر شراء | رفض، إلغاء |
| أمر شراء (ordered) | تأكيد كفاتورة | إلغاء |
| فاتورة (invoiced) | ترحيل (Post) | إلغاء |
| مرحّلة (posted) | تسجيل دفعة | — |

### المرحلة 4: صفحة القائمة (List/Kanban)

**لا تتغير كثيراً** — تبقى التبويبات كما هي:
- الكل | مسودة | طلبات | أوامر | فواتير | مرحّلة | مدفوعة
- زر الإنشاء يتغير نصه حسب التبويب النشط ✅ (تمت)
- الشيت يفتح بالحالة المناسبة ✅ (تمت)

### المرحلة 5: القواعد المحاسبية

| المرحلة | أثر محاسبي؟ | أثر مخزني؟ |
|---|---|---|
| request / quotation / order | ❌ لا | ❌ لا |
| invoiced | ❌ لا (فاتورة لم تُرحّل) | ❌ لا |
| posted | ✅ **نعم — قيد محاسبي** | يعتمد |
| paid | ✅ نعم | — |

## ⚠️ ملاحظات مهمة

### الفرق بين نهجنا ونهج Odoo:
- **Odoo**: الفاتورة (Vendor Bill) مستند منفصل `account.move` — لأن القيود المحاسبية في جدول مختلف
- **نهجنا**: الفاتورة **نفس المستند** — لكن القيد المحاسبي يُنشأ فقط عند `posted`
- **الميزة**: أبسط — مستند واحد فقط. لا حاجة لإدارة الربط بين مستندات
- **التحدي**: يجب التأكد أن المراحل المبكرة (request, quotation, order) لا تُنشئ أي أثر محاسبي

### ما الذي لا يتغير:
1. ✅ جداول قاعدة البيانات (`purchase_invoices`, `sales_invoices`) — تبقى كما هي
2. ✅ أعمدة `status` و `document_stage` — موجودة بالفعل
3. ✅ تريغر `update_document_stage()` — يعمل بالفعل
4. ✅ الحالات والتحولات في `custom_statuses` و `status_transitions` — موجودة

### ما الذي يتغير:
1. 🔄 `UnifiedTradeSheet` — إعادة بناء الـ header ليُظهر Status Bar
2. 🔄 `UnifiedAccountingSheet` — العنوان يتغير حسب المرحلة
3. 🔄 أزرار الإجراء — ديناميكية حسب الحالة الحالية
4. 🔄 إزالة مفهوم "نسخ المستند عند تغيير المرحلة"

### المرحلة 6: رقم المستند والـ QR Code

**الوضع الحالي:**
- الـ QR يأخذ `docNumber` من `data?.code || data?.entry_number || data?.id` — لا يستخدم `invoice_number`
- حقل الرقم المرجعي يعرض "AUTO-GEN" عند الإنشاء
- لا يوجد رقم واضح غير UUID

**المطلوب:**
1. ✅ رقم المستند (`invoice_number` / `order_number`) يظهر بوضوح في header الشيت
2. ✅ الـ QR يحتوي على رقم المستند الفعلي (وليس UUID)
3. ✅ الرقم يتغير prefix حسب المرحلة:
   - `PR-` للطلبات (Purchase Request)
   - `PQ-` لعروض السعر (Purchase Quotation)
   - `PO-` لأوامر الشراء (Purchase Order)
   - `PI-` لفواتير الشراء (Purchase Invoice)
   - `SR-` / `SQ-` / `SO-` / `SI-` للمبيعات
4. ✅ الرقم يكون `{PREFIX}-{YYYY}-{####}` (مثل: `PO-2026-0001`)
