## 🤔 السؤال: هل نحتاج مزامنة الأصناف؟

### ❌ ليس فوراً — وإليك السبب:

الجدولان **منفصلان بطبيعتهما**:

```
purchase_transaction_items     ≠     shipment_items
   (أصناف الفاتورة)                  (أصناف الكونتينر)
   
• ما طلبته وكم سعره          • ما يُشحن فعلياً في الكونتينر
• قد تشمل بضائع محلية         • فقط بضائع الكونتينر
• خاضعة لقيد محاسبي           • خاضعة للتتبع اللوجستي
• مرتبطة بمورد واحد           • قد تشمل عدة موردين
```

### ✅ ما نحتاجه فعلاً:
**الربط على مستوى الهيدر (الفاتورة ↔ الكونتينر) + قواعد عمل واضحة**

## 🏗️ التصميم المقترح — سير العمل الصحيح

### المبدأ الأساسي:
> **الفاتورة تُنهي عملها (ترحيل) ← ثم تُربط بالكونتينر ← ثم البضائع بعهدة الكونتينر**

### التسلسل الزمني:

```
┌──────────────────────────────────────────────────────┐
│                  دورة حياة الفاتورة                    │
│                                                       │
│  مسودة → عرض سعر → أمر شراء → فاتورة → ترحيل ✓      │
│  ↓                                                    │
│  [تعديل/حذف متاح]     [تعديل/حذف متاح]  [مقفلة]     │
└───────────────────────────────────┬──────────────────┘
                                    │
                          ┌─────────▼──────────┐
                          │   ربط بالكونتينر    │
                          │  (من صفحة الكونتينر) │
                          └─────────┬──────────┘
                                    │
                    ┌───────────────▼───────────────┐
                    │      البضائع بعهدة الكونتينر    │
                    │                                │
                    │  • الفاتورة مقفلة تماماً       │
                    │  • المصاريف على الكونتينر       │
                    │  • توزيع Landed Cost            │
                    │  • الاستلام من الكونتينر        │
                    └────────────────────────────────┘
```

## 📋 قواعد العمل (Business Rules)

### القاعدة 1: شروط الربط
| الشرط | السبب |
|-------|-------|
| الفاتورة يجب أن تكون **مرحّلة** (`stage = posted/partial_paid/paid`) | لضمان اكتمال بيانات الأصناف والأسعار |
| الفاتورة **غير مربوطة** بكونتينر آخر | منع التكرار |
| الكونتينر يجب أن يكون بحالة تسمح (ليس `received` أو `closed`) | لا يمكن الإضافة لكونتينر مُغلق |

### القاعدة 2: قفل الفاتورة عند الربط (الفواتير الكاملة)
| الإجراء | متاح؟ | السبب |
|---------|-------|-------|
| تعديل أصناف الفاتورة | ❌ ممنوع | البضائع بعهدة الكونتينر |
| تعديل أسعار الفاتورة | ❌ ممنوع | تؤثر على حسابات Landed Cost |
| حذف الفاتورة | ❌ ممنوع | مربوطة بكونتينر |
| إلغاء الترحيل | ❌ ممنوع | يخل بالتسلسل |
| عرض الفاتورة (read-only) | ✅ متاح | - |
| فك الارتباط من الكونتينر | ✅ متاح (بتأكيد) | يُعيد الفاتورة لحالة مرحّلة عادية |

### القاعدة 3: بعد فك الارتباط
- الفاتورة ترجع لحالة `posted` عادية (قابلة للتعديل أو إلغاء الترحيل)
- أصناف الكونتينر **لا تتأثر** (الأصناف في `shipment_items` مستقلة)
- يجب إعادة حساب Landed Cost على الكونتينر

## 🔮 حالة التحميل الجزئي (مؤجلة — Phase 2)

> **هذه حالة متقدمة يمكن العمل عليها لاحقاً بعد اكتمال الفواتير الكاملة**

### الوصف:
فاتورة واحدة فيها 5 أصناف → 3 أصناف تُحمّل بكونتينر A + 2 أصناف تُحمّل بكونتينر B

### التعقيدات:
```
┌──────────────────────────────┐
│        فاتورة شراء #1234      │
│                               │
│  صنف 1: قماش حرير 500م  ─────┬──→ كونتينر A  (محمّل ✅)
│  صنف 2: قماش قطن 300م   ─────┤
│  صنف 3: قماش كتان 200م  ─────┘
│                               │
│  صنف 4: بطانة 100م      ─────┬──→ كونتينر B  (محمّل ✅)
│  صنف 5: إكسسوار 50 قطعة ─────┘
│                               │
│  صنف 6: خيوط 20 علبة    ──────→  غير محمّل (متاح للتحميل)
└──────────────────────────────┘
```

### القواعد الخاصة (Phase 2):
- الأصناف **المحمّلة** ← مقفلة (لا تعديل)
- الأصناف **غير المحمّلة** ← قابلة للتعديل
- الفاتورة ككل ← جزئياً مقفلة (partial lock)
- يلزم عمود إضافي: `purchase_transaction_items.container_id` (FK)

### لماذا نؤجلها:
1. تتطلب UI معقد (قفل صف بصف)
2. تتطلب عمود جديد على مستوى أصناف الفاتورة
3. حالة نادرة — معظم الفواتير تُحمّل كاملة بكونتينر واحد

## 🚀 خطة التنفيذ المقترحة — Phase 3 المحدّثة

### Phase 3A: ربط الفاتورة الكاملة بالكونتينر ⏱️ ~90 دقيقة

#### المهمة 3A.1: واجهة الربط (من صفحة الكونتينر)
**الموقع:** صفحة إدارة الكونتينر → تبويب "الفواتير"  
**العمل:**
- [ ] زر "ربط فاتورة" → يفتح dialog لاختيار فاتورة مرحّلة
- [ ] فلترة: فقط الفواتير بحالة `posted/partial_paid/paid` + غير مربوطة
- [ ] عرض: رقم الفاتورة + المورد + المبلغ + تاريخ الترحيل
- [ ] عند الاختيار: 
  - تحديث `purchase_transactions.container_id`
  - نسخ أصناف الفاتورة إلى `shipment_items` (مع `purchase_invoice_id`)

#### المهمة 3A.2: قفل الفاتورة بعد الربط
**الموقع:** `UnifiedAccountingSheet` + `tradeConfigs.ts`  
**العمل:**
- [ ] إضافة حالة جديدة: `container_locked` 
- [ ] عند فتح فاتورة مربوطة بكونتينر → إخفاء أزرار التعديل
- [ ] إظهار شريط تنبيه: "⚓ هذه الفاتورة مربوطة بالكونتينر [رقم] — للتعديل أفصل الارتباط"
- [ ] زر "فك الارتباط" على الشريط (بتأكيد)

#### المهمة 3A.3: واجهة فك الارتباط
**العمل:**
- [ ] تأكيد: "هل تريد فك ارتباط الفاتورة من الكونتينر؟ سيتطلب إعادة حساب Landed Cost"
- [ ] تفريغ `container_id` من الفاتورة
- [ ] حذف الأصناف المنسوخة من `shipment_items` (حيث `purchase_invoice_id = ?`)
- [ ] إعادة حساب تكاليف الكونتينر

## 📊 ملخص تنفيذي

| الجانب | القرار |
|--------|--------|
| **متى تُربط الفاتورة؟** | بعد الترحيل فقط |
| **من أين؟** | من صفحة الكونتينر (ليس من الفاتورة) |
| **هل الفاتورة قابلة للتعديل بعد الربط؟** | ❌ لا — مقفلة بالكامل |
| **كيف تُعدّل؟** | فك الارتباط أولاً ← تعديل ← إعادة ربط |
| **التحميل الجزئي؟** | Phase 2 — نبدأ بالفواتير الكاملة |
| **أين المصاريف؟** | على الكونتينر فقط (`shipment_costs`) |
| **أين البضائع بعد الربط؟** | بعهدة الكونتينر (`shipment_items`) |
